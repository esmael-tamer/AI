# مراجعة مشروع Media Trend - منصة التجارة الإلكترونية بالذكاء الاصطناعي

## نظرة عامة على المشروع

**Media Trend** هي منصة تجارة إلكترونية مدعومة بالذكاء الاصطناعي تعتمد نموذج "جرّب أولاً"، حيث يمكن للمستخدمين إنشاء متاجر تجريبية عبر محادثة ذكية دون الحاجة للتسجيل، ثم التسجيل وطلب تفعيل الخدمات (المدفوعات، الشحن، التخزين).

- **نموذج الإيرادات**: عمولة فقط (بدون اشتراكات)
- **الأسواق المستهدفة**: الكويت والسعودية (جاهز للتوسع العالمي)
- **اللغات**: عربي (RTL) + إنجليزي

## نقاط القوة

### 1. بنية منظمة جيداً
- استخدام Next.js App Router بشكل صحيح مع فصل واضح بين الصفحات والمكونات وواجهات API
- نظام أنواع TypeScript شامل لقاعدة البيانات (`lib/db.ts`)
- مكتبة مكونات غنية (75+ مكون) مبنية على Radix UI

### 2. دعم ثنائي اللغة متكامل
- نظام i18n مبني على React Context مع دعم كامل لـ RTL
- حفظ تفضيل اللغة في localStorage
- جميع المكونات تدعم دالة الترجمة `t(en, ar)`

### 3. استعلامات SQL آمنة
- جميع الاستعلامات تستخدم Parameterized Queries عبر مكتبة `postgres`
- لا يوجد أي تجميع نصي مباشر لاستعلامات SQL (حماية من SQL Injection)

### 4. محرك بناء المتاجر بالذكاء الاصطناعي
- تدفق محادثة من 7 خطوات مع ردود ثنائية اللغة
- توليد تلقائي لمنتجات تجريبية حسب الفئة
- إنشاء روابط فريدة (slugs) للمتاجر

### 5. نظام إدارة شامل
- لوحة تحكم إدارية مع 11 قسم
- بوابة عملاء مع stepper لتفعيل الخدمات
- سجل تدقيق (Audit Log) لتتبع النشاطات الإدارية

## مشاكل حرجة (يجب إصلاحها فوراً)

### 1. كلمة مرور مبرمجة في الكود
**الملف**: `app/api/auth/login/route.ts`
**الخطورة**: حرجة

```typescript
const isValid = password === "admin123" || ...
```

كلمة المرور `admin123` مقبولة لأي عملية تسجيل دخول، مما يسمح بالوصول غير المصرح به لأي حساب.

**التوصية**: إزالة كلمة المرور المبرمجة واستخدام التحقق من كلمة المرور المشفرة فقط.

---

### 2. تخزين كلمات المرور كنص صريح
**الملف**: `app/api/auth/signup/route.ts`
**الخطورة**: حرجة

```typescript
INSERT INTO users (..., password_hash, ...)
VALUES (..., ${password}, ...)
```

كلمة المرور تُخزن كنص صريح بدون تشفير رغم أن الحقل يسمى `password_hash`. دالة `hashPassword` الموجودة في `lib/auth.ts` لا تُستخدم أثناء التسجيل.

**التوصية**: استخدام `hashPassword()` من `lib/auth.ts` قبل التخزين، أو الأفضل استخدام `bcrypt` أو `Argon2`.

---

### 3. عدم وجود مصادقة على واجهات API الإدارية
**الخطورة**: حرجة

جميع مسارات `/api/admin/*` (11 مسار) لا تحتوي على أي تحقق من المصادقة:
- `GET/POST /api/admin/users`
- `GET/POST /api/admin/stores`
- `GET/POST /api/admin/leads`
- `GET/POST /api/admin/tickets`
- `GET/POST /api/admin/blog`
- `GET/POST /api/admin/pages`
- `GET/POST /api/admin/team`
- `GET/POST /api/admin/partners`
- `GET/POST /api/admin/cases`
- `GET /api/admin/stats`
- `GET /api/admin/audit`

الـ middleware يحمي صفحات الواجهة فقط وليس نقاط API مباشرة. أي شخص يمكنه الوصول لهذه البيانات عبر استدعاء API مباشر.

**التوصية**: إضافة `requireAdmin()` من `lib/auth.ts` في بداية كل دالة API إدارية.

---

### 4. خوارزمية تشفير كلمات المرور ضعيفة
**الملف**: `lib/auth.ts:16`
**الخطورة**: عالية

```typescript
const data = encoder.encode(password + "mediatrend-salt-2024")
const hash = await crypto.subtle.digest("SHA-256", data)
```

- استخدام SHA-256 غير مناسب لتشفير كلمات المرور (سريع جداً = قابل للكسر)
- Salt ثابت ومبرمج في الكود
- لا يوجد عامل تكلفة (cost factor)

**التوصية**: استخدام `bcrypt` مع cost factor 12+ أو `Argon2id`.

## مشاكل عالية الخطورة

### 5. عدم وجود Rate Limiting
**الخطورة**: عالية

لا يوجد أي تحديد لعدد الطلبات على أي نقطة API:
- `/api/auth/login` معرضة لهجمات القوة الغاشمة (Brute Force)
- `/api/auth/signup` معرضة لإنشاء حسابات وهمية بكميات كبيرة
- `/api/leads` معرضة لهجمات الإغراق (Spam)

**التوصية**: إضافة rate limiting باستخدام `next-rate-limit` أو حل مشابه.

---

### 6. عدم التحقق من صحة المدخلات
**الخطورة**: عالية

مسارات متعددة تقبل مدخلات المستخدم بدون تحقق كافٍ:
- لا يوجد تحقق من صيغة البريد الإلكتروني (RFC 5322)
- `commission_rate_percent` يُقبل بدون التحقق من النوع أو النطاق
- `payload_json` يُقبل بدون تحقق قبل `JSON.stringify()`
- `selected_activations` يقبل أي مصفوفة بدون فلترة

**التوصية**: استخدام مكتبة `zod` (موجودة أصلاً في المشروع) للتحقق من جميع المدخلات.

---

### 7. عدم وجود اختبارات
**الخطورة**: عالية

- لا يوجد أي ملفات اختبار (`.test.ts`, `.spec.ts`)
- لا يوجد إعداد لـ Jest أو Vitest
- لا يوجد سكريبت اختبار في `package.json`

**التوصية**: إضافة إطار اختبار (Vitest مع Next.js) وكتابة اختبارات للمنطق الحساس (المصادقة، API endpoints، محرك البناء).

## مشاكل متوسطة الخطورة

### 8. الـ Middleware لا يتحقق من صلاحية الجلسة
**الملف**: `middleware.ts`

- يتحقق فقط من وجود الكوكي وليس من صلاحيتها
- لا يتحقق من دور المستخدم (أي مستخدم مسجل يمكنه الوصول لصفحات الإدارة)
- لا يتحقق من انتهاء صلاحية الجلسة

---

### 9. تعطيل ESLint و TypeScript في البناء
**الملف**: `next.config.mjs`

```javascript
eslint: { ignoreDuringBuilds: true },
typescript: { ignoreBuildErrors: true },
```

هذا يعني أن أخطاء الأنواع وأخطاء الكود لا تُكتشف أثناء البناء.

**التوصية**: تفعيل فحص ESLint و TypeScript والتعامل مع الأخطاء الموجودة.

---

### 10. استخدام `dangerouslySetInnerHTML`
**الملفات**:
- `components/ui/chart.tsx:74`
- `app/3D-architecture-visualization-studio/page.tsx:53`
- `app/About/page.tsx:44`

يجب التأكد من أن المحتوى المُمرر لا يأتي من مدخلات المستخدم.

---

### 11. توليد Session ID غير آمن تشفيرياً
**الملف**: `app/api/stores/route.ts:39`

```typescript
const sessionId = `session_${Date.now()}_${Math.random().toString(36).substr(2, 9)}`
```

`Math.random()` ليست آمنة تشفيرياً.

**التوصية**: استخدام `crypto.randomUUID()`.

**الملف**: `package.json`

- `vue-router: latest` - اعتماد غير مستخدم (مشروع React وليس Vue)

**التوصية**: إزالتها أو استبدالها بنظام تسجيل منظم (structured logging).

---

### 14. عدم وجود CI/CD

- لا يوجد pre-commit hooks

- `X-Frame-Options`
